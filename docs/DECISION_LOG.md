## 결정 기록

### 2026-07-07 · 요금제 · 콘텐츠 등록 할당량 (Billing / Quota)

**배경**
상용화의 핵심인 "매출" 기반을 만들기 위해 요금제와 콘텐츠 등록 한도를 도입. 이 시점에 실제 결제(PG)는 붙이지 않고, 요금제/할당량 뼈대를 먼저 세우기로 함.

**요금제 (기본값 — 관리자가 변경 가능)**

| 요금제 | 가격 | 콘텐츠 한도 | 주기 |
|--------|------|-------------|------|
| free | 0원 | 1개 | **평생** |
| basic | 월 3,000원 | 5개 | 매월 |
| premium | 월 5,000원 | 10개 | 매월 |
| max | 월 7,000원 | 20개 | 매월 |

**정책 결정 3가지와 그 이유(철학)**

1. **결제 범위: "요금제·할당량 먼저, 실결제(PG)는 나중"**
   - *이유*: 실제 PG 연동은 결제사 가입·API 키·웹훅이 필요해 외부 의존성이 크다. 그 전에 요금제 데이터 모델과 한도 강제 로직을 완성해 두는 것이 순서상 옳다.
   - *현재 운영 방식*: 관리자가 `/admin`에서 사용자 요금제를 **수동 부여**한다. (PG가 붙으면 이 수동 부여를 결제 성공 이벤트가 대신하게 된다.)
   - *기각한 대안*: 지금 바로 토스페이먼츠/Stripe 연동 — 아직 결제사 미정, 키 없음.

2. **카운트 기준: "월 신규 등록 수" + "삭제해도 복구 안 됨"**
   - *이유(철학)*: 한도는 "지금 보유 중인 개수"가 아니라 **"그 주기에 새로 등록한 횟수"** 로 센다. 콘텐츠를 등록했다가 지우고 다시 등록하는 식으로 한도를 무한히 우회하는 것을 막기 위함이다. 등록은 실제 비용(자막 처리·AI 호출 등)을 유발하는 행위이므로, 등록 그 자체를 과금 단위로 본다.
   - *설계적 함의(중요)*: "삭제해도 카운트가 복구되면 안 된다"는 요구는 **영상 행(row)을 세면 안 된다**는 뜻이다(영상을 지우면 행도 사라지므로). 그래서 별도의 **등록 원장(ledger) 테이블** `content_registrations` 를 두고, 영상이 삭제돼도 원장 기록은 남긴다.
   - *멱등성*: 같은 콘텐츠를 (지우지 않은 채) 다시 등록하면 카운트되지 않는다. 원장의 `(user_id, video_id)` UNIQUE로 보장. 단, **지웠다가 다시 등록하면** 새 영상 id가 생기므로 새 등록으로 카운트된다(정책상 옳음).
   - *기각한 대안*: "동시 보유 수 기준"(삭제하면 슬롯 복구) — 우회가 쉽고, 등록 비용을 반영하지 못함.

3. **월 리셋 기준: "구독 시작일 기준"**
   - *이유*: 실제 결제 주기와 정확히 일치시키기 위함(가입/결제한 날부터 한 달). 달력 1일 기준은 구현이 단순하지만 결제 주기와 어긋난다.
   - *구현*: `plan_started_at`(구독 시작일)의 "가장 최근 매월 기념일"을 계산해 그 이후 등록 수를 센다. 짧은 달은 말일로 보정. 관리자가 요금제를 바꾸면 `plan_started_at`이 그 시점으로 갱신된다.

4. **free는 "평생 1개"의 철학**
   - free는 매월 리셋되지 않는다(`period_type='lifetime'`). 무료 사용자는 **평생 총 1개**만 등록할 수 있다. 맛보기 체험용이며, 계속 쓰려면 유료 전환을 유도하는 구조.

5. **가격·한도는 코드가 아니라 DB에**
   - 요금제 값을 하드코딩하지 않고 `plans` 테이블에 저장해, **관리자 도구에서 즉시 변경**할 수 있게 했다. 가격 정책은 시장 반응에 따라 자주 바뀔 수 있으므로 배포 없이 조정 가능해야 한다는 철학.

**한도 초과 시 동작**
등록 시 한도를 넘으면, 방금 만든 콘텐츠를 **롤백(삭제)** 하고 `HTTP 402 {error:"quota_exceeded", message, usage}` 를 반환한다. 프론트는 안내 문구를 토스트로 띄우고 사이드바 사용량 배지를 갱신한다.

**구현 요약 (파일별)**
- `database.py`
  - `plans` 테이블 + 기본 4단계 시드, `content_registrations` 원장(등록 이벤트)
  - `users` 컬럼 추가: `plan_code`(기본 free), `plan_started_at`(구독 시작일)
  - 마이그레이션 시 기존 영상을 원장에 **백필**(등록일=영상 생성일). `claim_orphan_data`(첫 가입자 데이터 인수)에서도 원장 백필
  - 함수: `get_plans/get_plan/update_plan`, `set_user_plan`, `get_usage`(주기별 사용량 계산), `record_content_registration`(멱등), `undo_content_registration`(롤백), `_period_start`(구독일 기준 주기 계산)
- `server.py`
  - `_finalize_registration(video_id)` — 콘텐츠 5개 등록 경로(YouTube·가사·텍스트·URL·파일) 공통 choke point. 원장 기록 + 한도 강제 + 초과 시 롤백/402
  - 관리자 API: `GET /api/admin/plans`, `PUT /api/admin/plans/<code>`(가격·한도), `PUT /api/admin/users/<id>/plan`(요금제 부여). `admin_list_users`에 요금제/사용량 포함
  - 사용자 API: `GET /api/me/plan`(내 사용량 + 요금제 목록)
- `templates/admin.html` — 요금제 편집 카드(가격·한도 저장) + 사용자 표에 요금제 드롭다운·사용량 pill
- `templates/index.html` — 사이드바 요금제/사용량 배지(`refreshPlanBadge`), 402 안내 문구 정규화(`api()` 헬퍼)

**알려진 한계 / 향후 결정 필요**
- **플레이리스트 자동 동기화**(백그라운드 워커)로 추가되는 영상은 현재 할당량 강제 대상이 **아니다**. 요청 컨텍스트가 없고 자동 추가라 블로킹이 부적절해 이번 범위에서 제외. → 정책(자동 추가분을 카운트할지/블록할지) 확정 후 반영 필요.
- **실결제(PG) 미연동**. 지금은 관리자 수동 부여.
- 동시 요청 레이스: 극히 짧은 순간 두 요청이 동시에 한도 검사를 통과해 1개 초과 등록될 수 있음(단일 사용자 통상 사용에선 사실상 무해). 필요 시 등록 시 트랜잭션 잠금 추가.

**상태**: ✅ 구현·검증 완료. 로직/웹(E2E 402 흐름·롤백)/렌더 자동 테스트 통과.

### 2026-07-06 · 관리자 운영 도구 (`/admin`)
- **무엇**: 전체 사용자 목록(데이터량·로그인수단·최근활동) + 요약 통계 대시보드.
- **왜**: 상용 운영에는 누가 얼마나 쓰는지 볼 수단이 필요.
- **정책**: 관리자 판별 = `ADMIN_EMAILS` 환경변수(콤마 구분), 미설정 시 첫 사용자(id=1). 비밀번호 해시·AI 키 값은 절대 노출 안 함. 관리자 조회는 `_uid()` 격리 필터를 우회하므로 반드시 서버측 `is_admin` 검증 후에만 허용.

### 2026-07-06 · Step 3 · 구글 OAuth + 사용자별 AI 키
- **왜 OAuth**: 이메일/비번 없이 간편 로그인 제공.
- **철학(의존성 최소)**: 외부 라이브러리 없이 **표준 라이브러리(urllib)만** 으로 OAuth 구현. 미설정(키 없음) 시 구글 버튼 자동 비활성.
- **AI 키 정책(둘 다 지원)**: 사용자 개인 키가 있으면 그것을, 없으면 **서버 공용 키로 폴백**. 배경 워커(전역 캐시 생성)는 서버 키, 사용자 상호작용은 개인 키 우선.

### 2026-07-06 · Step 2 · 멀티유저 데이터 완전 격리
- **무엇**: 7개 테이블에 `user_id` 부여 + 복합 UNIQUE 재구축(같은 콘텐츠도 사용자별 독립 보관).
- **철학(격리 방식)**: 라우트마다 조건을 다는 대신, `before_request`에서 `set_current_user()`로 스레드-로컬 컨텍스트를 설정하고 모든 DB 함수가 `_uid()`로 **자동 필터**. 실수로 격리를 빠뜨릴 여지를 구조적으로 차단.
- **기존 데이터**: 첫 가입자가 `claim_orphan_data()`로 소유자 없는 기존 단일 사용자 데이터를 인수.
- **전역 공유 유지**: `word_meanings`/`ai_cache`는 내용 기반 캐시라 사용자 공유(비용 절감).

### 2026-07-06 · Step 1 · 이메일/비밀번호 인증 + 클라우드 배포
- **인증**: `pbkdf2:sha256` 해싱, Flask 서명 세션, `before_request` 전역 인증 게이트.
- **배포 철학**: 로컬 전용 앱을 환경변수 기반으로 전환(`DATA_DIR`, `SECRET_KEY` 등). 프로덕션은 `gunicorn`, `init_app()`이 import 시점에 DB·백그라운드 스레드 시작. 배포 대상 Fly.io(도쿄), 영구 볼륨에 SQLite.

## 전반 설계 철학 (프로젝트 공통)

- **의존성 최소주의**: 가능하면 표준 라이브러리만. (OAuth도 urllib으로 구현)
- **SQLite 단일 서버 전제**: 단순함 우선. 스케일아웃이 필요해지면 PostgreSQL 전환(문서 ARCHITECTURE 11.3에 명시). 지금 최적화하지 않는다.
- **문서 자동 갱신 규칙**: 코드를 고치면 `USER_MANUAL`/`ARCHITECTURE`/`FEATURE_SPEC` 3종을 함께 갱신(`CLAUDE.md`). 이 `DECISION_LOG`는 그 위에 **"왜"** 를 더한다.
- **구조로 실수를 막는다**: 개별 호출부에 방어 코드를 흩뿌리기보다, 공통 choke point/컨텍스트로 강제(예: 데이터 격리의 `_uid()`, 할당량의 `_finalize_registration`).
- **정책 값은 데이터로**: 자주 바뀌는 값(요금제 가격·한도)은 코드가 아니라 DB에 두어 배포 없이 조정.

## 용어

- **콘텐츠(content) / 영상(video)**: 학습 단위. 내부 테이블명은 `videos`지만 YouTube·텍스트·URL·파일·가사를 모두 포함한다.
- **등록(registration)**: 콘텐츠를 받아와 **전체학습/개별학습이 가능한 상태**가 되는 것. 할당량 카운트의 기준 시점.
- **원장(ledger)**: `content_registrations` 테이블. 등록 이벤트를 영구 기록(영상 삭제와 무관).
- **주기(period)**: 할당량이 리셋되는 단위. free=평생(lifetime), 유료=구독 시작일 기준 매월(monthly).
